# Plan: Персонализация данных и управление доступом

**Статус: ✅ ВЫПОЛНЕНО (2026-07-09)**

## Проблема

При заходе как "Гость" все задачи пропадают для всех пользователей. Причины:
1. Guest — один общий пользователь, у него свой workspace, но данные в `Workspace.allData` хранятся как JSON-строка
2. Вопросы глобальные (нет привязки к workspace)
3. Нет контроля видимости отдельных задач
4. Нет возможности запросить доступ к чужому workspace

## Решение — 4 фазы

---

### Фаза 1: Гостевой доступ — универсальный просмотр

**Цель**: Гость — один общий пользователь, который видит **все задачи** всех workspace без исключения, но не может их редактировать.

#### 1.1 Изменить `src/app/api/auth/guest/route.ts`

Гость — **всегда один и тот же пользователь**:
```
username: "guest"
displayName: "Гость"
```
При входе как гость — создаётся (или находит существующего) пользователя `guest` с одним workspace. Все гости работают в одном workspace — данные общие.

#### 1.2 Guest — read-only доступ

Гость имеет доступ ко всем задачам всех workspace, но:
- **Не может** создавать, редактировать, удалять задачи
- **Не может** менять статусы, приоритеты, оценки
- **Не может** оставлять комментарии
- **Может** просматривать всё без ограничений (visibleTo не применяется)

Реализация: на клиенте блокировать все write-операции для role="guest". На сервере — гость не проходит проверкуwrite-доступа в sync/API.

#### 1.3 Добавить в `prisma/schema.prisma` поле `visibleTo` в Workspace

```prisma
model Workspace {
  ...
  visibleTo String @default("[]")  // JSON array user IDs — кто видит workspace
}
```

Это пока заглушка для будущей фазы 3.

### Фаза 2: Таблица задач в Supabase (per-task storage)

**Цель**: Каждая задача — отдельная строка в БД, привязанная к workspace.

#### 2.1 Новая таблица `Task` в `prisma/schema.prisma`

```prisma
model Task {
  id           String   @id @default(cuid())
  workspaceId  String
  workspace    Workspace @relation(fields: [workspaceId], references: [id], onDelete: Cascade)
  domainId     String   @default("default")
  monthKey     String   // "YYYY-MM" или "backlog"
  num          String   @default("")
  name         String   @default("")
  planH        String   @default("0")
  factH        String   @default("0")
  priority     String   @default("QUEUE")
  status       String   @default("IDEA")
  comment      String   @default("")
  commentLog   String   @default("[]")  // JSON array
  visibleTo    String   @default("[]")  // JSON array user IDs — кто видит задачу
  _ts          DateTime @default(now())
  _deleted     Boolean  @default(false)
  createdAt    DateTime @default(now())
  updatedAt    DateTime @updatedAt

  @@index([workspaceId])
  @@index([workspaceId, domainId])
  @@index([workspaceId, domainId, monthKey])
}
```

#### 2.2 Обновить `Workspace` — связь с задачами

```prisma
model Workspace {
  ...
  tasks      Task[]
  questions  Question[]  // добавить связь
}
```

#### 2.3 Обновить `Question` — привязка к workspace

```prisma
model Question {
  ...
  workspaceId String?
  workspace   Workspace? @relation(fields: [workspaceId], references: [id], onDelete: Cascade)

  @@index([workspaceId])
}
```

#### 2.4 Миграция существующих данных

Скрипт миграции (`prisma/migrate-task-storage.ts`):
1. Читает все `Workspace.allData` (JSON)
2. Парсит `Record<domainId, DomainData>` → извлекает задачи
3. Создаёт строки в `Task` таблице
4. Привязывает вопросы к workspace (если workspaceId null — оставляем глобальными)
5. Помечает миграцию как завершённую

### Фаза 3: API маршруты для per-task CRUD

**Цель**: CRUD задач через REST, с фильтрацией по visibleTo.

#### 3.1 Новый `src/app/api/tasks/route.ts`

```
GET  /api/tasks?workspaceId=...&token=...
  → Возвращает задачи workspace, фильтруя по visibleTo
  → Если user в owner/editor — видит все
  → Если viewer/executive — видит только те, где visibleTo пустой ИЛИ содержит userId

POST /api/tasks?token=...
  → Создаёт задачу
  → Автоматически добавляет ownerId в visibleTo

PUT  /api/tasks/:id?token=...
  → Обновляет задачу
  → Только owner/editor/executive

DELETE /api/tasks/:id?token=...
  → Soft delete (sets _deleted=true)
```

#### 3.2 Новый `src/app/api/tasks/bulk/route.ts`

```
POST /api/tasks/bulk?token=...
  → Массовое обновление (для sync)
  → Принимает массив задач, применяет last-write-wins
```

#### 3.3 Обновить `src/app/api/sync/route.ts`

Добавить поддержку per-task sync:
- GET: возвращать задачи из `Task` таблицы (с фильтрацией по visibleTo)
- POST: принимать задачи и сохранять в `Task` таблицу
- Обратная совместимость: если клиент отправляет `domainData` (старый формат) — конвертировать в per-task

#### 3.4 Обновить `src/app/api/question/route.ts`

- GET: фильтровать по `workspaceId` (если передан)
- POST: привязывать к `workspaceId`
- PUT/DELETE: проверять доступ к workspace

### Фаза 4: UI изменения

**Цель**: Обновить интерфейс для работы с новой архитектурой.

#### 4.1 Обновить `src/components/dialogs/share-dialog.tsx`

- Заменить ручной ввод имени пользователя на **выпадающий список**
- Добавить API `GET /api/users?token=...` для получения списка пользователей
- Добавить кнопку "Запросить доступ" для пользователей без доступа

#### 4.2 Новый `src/app/api/users/route.ts`

```
GET /api/users?token=...
  → Возвращает список всех активных пользователей
  → Только для авторизованных
```

#### 4.3 Новый `src/app/api/workspace/requests/route.ts`

```
GET  /api/workspace/requests?workspaceId=...&token=...
  → Получить входящие запросы на доступ (для owner)

POST /api/workspace/requests?token=...
  → Отправить запрос на доступ к workspace

PUT  /api/workspace/requests/:id?token=...
  → Одобрить/отклонить запрос (для owner)
```

#### 4.4 Новая таблица `WorkspaceAccessRequest`

```prisma
model WorkspaceAccessRequest {
  id          String    @id @default(cuid())
  workspaceId String
  workspace   Workspace @relation(fields: [workspaceId], references: [id], onDelete: Cascade)
  userId      String
  user        User      @relation(fields: [userId], references: [id], onDelete: Cascade)
  status      String    @default("pending")  // pending | approved | rejected
  message     String    @default("")
  createdAt   DateTime  @default(now())
  updatedAt   DateTime  @updatedAt

  @@unique([workspaceId, userId])
  @@index([workspaceId])
  @@index([userId])
}
```

#### 4.5 Обновить `src/lib/store.ts`

- Добавить action `setVisibility(taskId, userIds)` — обновляет `visibleTo` задачи
- При загрузке задач с сервера — фильтровать по visibleTo на клиенте (дополнительная защита)

#### 4.6 Обновить `src/hooks/useServerSync.ts`

- При pull: загружать задачи из `/api/tasks` (per-task endpoint)
- При push: отправлять изменённые задачи через `/api/tasks/bulk`
- Обратная совместимость: поддерживать старый формат `domainData` до полного перехода

## Верификация

1. Войти как гость → видит все задачи всех пользователей
2. Гость не может создать/отредактировать/удалить задачу (UI заблокирован, API возвращает ошибку)
3. Создать задачу как пользователь A → пользователь B её не видит
4. Поделиться workspace → пользователь B видит задачи A
5. Скрыть конкретную задачу → пользователь B её не видит даже в shared workspace
6. Запросить доступ → owner видит запрос и может одобрить
7. Вопросы привязаны к workspace → у разных workspace разные вопросы
