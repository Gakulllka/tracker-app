# Plan: Executive Access — Domain-Level Workspace Sharing

## Goal
Руководители не нуждаются в личном workspace. Аналитик назначает каждому руководителю конкретные домены в своём workspace. При входе руководитель автоматически попадает в нужный workspace и видит только назначенные домены. Руководитель может просматривать задачи и оставлять комментарии/executive flags, но не менять статусы.

## Current State (уже реализовано)
- `WorkspaceShare` модель: workspaceId + userId + role ("editor"|"viewer")
- ShareDialog: добавление пользователей, смена роли, отзыв доступа
- Sidebar workspace switcher: переключение между своими и общими workspace'ами
- Sync API: access control по workspace (read/write)

## What Needs to Change

### 1. Database — добавить поле `domainIds` в WorkspaceShare

**File:** `prisma/schema.prisma`

```prisma
model WorkspaceShare {
  id          String    @id @default(cuid())
  workspaceId String
  workspace   Workspace @relation(fields: [workspaceId], references: [id], onDelete: Cascade)
  userId      String
  user        User      @relation(fields: [userId], references: [id], onDelete: Cascade)
  role        String    @default("editor")  // "editor" | "viewer" | "executive"
  domainIds   String    @default("[]")      // JSON array of domain IDs — какие домены видит руководитель
  createdAt   DateTime  @default(now())

  @@unique([workspaceId, userId])
  @@index([userId])
}
```

Роль `"executive"` — руководитель. Видит только назначенные домены, может комментировать и ставить флаги, но не менять статусы.

### 2. API Changes

**`POST /api/workspace/shares`** — добавить поле `domainIds`:
- Body: `{ token, workspaceId, username, role, domainIds?: string[] }`
- При роли `"executive"` — `domainIds` обязателен

**`PUT /api/workspace/shares`** — обновлять и `domainIds`:
- Body: `{ token, shareId, role, domainIds?: string[] }`

**`GET /api/workspace/shares`** — возвращать `domainIds`:
- Response: `{ shares: [{ id, userId, username, displayName, role, domainIds }] }`

**`GET /api/auth/me`** — добавить `domainIds` в `accessibleWorkspaces`:
```typescript
accessibleWorkspaces: [{
  workspaceId, name, role, ownerName,
  domainIds: string[]  // NEW: какие домены видит руководитель
}]
```

### 3. Auth Flow — Auto-Redirect for Executives

**`GET /api/auth/me`** — если пользователь имеет role="executive" в WorkspaceShare:
- Автоматически переключать `workspaceId` на shared workspace
- Возвращать `domainIds` чтобы фронтенд мог ограничить видимость

**`src/hooks/useAuth.ts`** — `switchWorkspace` обновлён:
- При переключении на shared workspace с domainIds → store автоматически показывает только эти домены

### 4. Frontend: ShareDialog — Domain Assignment

**`src/components/dialogs/share-dialog.tsx`** — расширить:

При выборе роли "executive" показывается чекбокс-лист доменов:
```
[✓] Фронтенд
[ ] Бэкенд
[✓] Мобилка
```

Домены загружаются из текущего workspace (store.domains). При сохранении — `domainIds` отправляется на сервер.

### 5. Frontend: Domain Visibility for Executives

**`src/hooks/usePermissions.ts`** — добавить логику:

Если пользователь — executive в текущем workspace:
- `visibleDomains` = только домены из `domainIds`
- `canEdit` = false (не может менять статусы)
- `canComment` = true (может оставлять комментарии)
- `canSetFlags` = true (может ставить executive flags)

Нужно передать в usePermissions информацию о текущей share-записи.

**`src/app/page.tsx`** — передать в usePermissions:
- Найти текущую share-запись для workspaceId из authData.accessibleWorkspaces
- Передать domainIds и role в usePermissions

### 6. Frontend: Hide Workspace Switcher for Executives

Если у пользователя только одна роль "executive" и нет своего workspace — не показывать секцию "Пространства" в сidebar (авто-редирект и так работает). Если есть и свой workspace, и executive доступ — показывать переключатель.

### 7. Sync API — Executive Write Rules

**`POST /api/sync`** — для executive:
- Разрешить пуш данных (для комментариев и флагов)
- Но на клиенте блокировать изменение статусов через UI
